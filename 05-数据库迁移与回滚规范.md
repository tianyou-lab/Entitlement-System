# 数据库迁移与回滚规范

更新时间：2026-05-14

## 1. 原则

- 生产环境只执行 Prisma migration，不直接执行 `db push`。
- 每次 schema 变更必须生成迁移文件并随代码一起提交。
- 执行生产迁移前必须完成数据库备份。
- 迁移文件进入主分支前必须人工审核 SQL。
- 破坏性变更必须提供数据保留、回填或回滚方案。

## 2. 开发流程

修改 `server/prisma/schema.prisma` 后生成迁移：

```bash
npm --prefix server exec -- prisma migrate dev --schema server/prisma/schema.prisma
```

提交前校验：

```bash
npm --prefix server exec -- prisma validate --schema server/prisma/schema.prisma
npm --prefix server exec -- prisma migrate diff --from-empty --to-schema-datamodel server/prisma/schema.prisma --script --output /tmp/entitlement-schema.sql
npm --prefix server run test:e2e
```

必须提交：

- `server/prisma/schema.prisma`
- `server/prisma/migrations/*/migration.sql`
- 相关服务代码、测试和文档

## 3. 迁移审核清单

审核 `migration.sql` 时重点看：

1. 是否包含 `DROP TABLE`、`DROP COLUMN`、`TRUNCATE` 等破坏性语句。
2. 是否对大表增加非空字段但没有默认值或回填策略。
3. 是否新增唯一索引，且历史数据可能重复。
4. 是否修改 enum，影响旧数据或旧客户端。
5. 是否新增外键，历史数据是否满足约束。
6. 是否需要先发布兼容代码，再执行迁移。
7. 是否需要拆分为“加字段、回填、切流、删旧字段”多步迁移。

## 4. 生产执行流程

1. 确认当前镜像和代码 commit。
2. 执行备份：

```bash
scripts/prod-backup.sh
```

3. 执行迁移：

```bash
scripts/prod-migrate.sh
```

4. 执行健康检查：

```bash
scripts/prod-healthcheck.sh
```

5. 查看日志：

```bash
scripts/prod-logs.sh server
```

6. 登录 Admin 抽查核心数据。

## 5. 回滚策略

代码回滚：

1. 保留上一版镜像或 tag。
2. 切回上一版镜像或代码。
3. 重启服务。
4. 执行健康检查。

数据库回滚：

- Prisma migration 默认不自动生成 down migration。
- 对无破坏性迁移，优先通过兼容代码回滚。
- 对破坏性迁移，必须在执行前准备恢复备份方案。
- 需要恢复备份时，先在独立测试环境演练，再决定是否覆盖生产库。

恢复备份参考：

```bash
BACKUP_FILE=backups/entitlement-YYYYmmdd-HHMMSS.sql
docker compose --env-file .env.production -f docker-compose.prod.yml exec -T postgres psql -U entitlement -d entitlement < "$BACKUP_FILE"
```

## 6. 试运行要求

试运行期间每次迁移都记录：

- 执行日期。
- 执行人。
- 代码 commit。
- 迁移目录名。
- 备份文件路径。
- 迁移耗时。
- 健康检查结果。
- 是否需要回滚。
