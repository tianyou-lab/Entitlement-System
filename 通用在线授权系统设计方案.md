# 通用在线授权系统设计方案

## 1. 目标

设计一套 **可复用、可商业化、可接入多个桌面程序** 的通用授权系统。  
该系统不针对某一个现有程序，而是作为独立产品存在，后续可以接入：

- Electron 桌面程序
- C/C++ 原生程序
- .NET 程序
- Python 打包程序
- 其他 Windows 桌面软件

系统需要覆盖以下能力：

- 授权 UI
- 在线激活
- 在线验证
- 设备绑定
- 到期控制
- 功能位控制
- 远程封禁
- 版本控制
- 本地缓存与加密
- SDK 接入
- 加壳/保护流程接入
- 管理后台
- 渠道/代理商/卡密或 License 管理

---

## 2. 设计原则

### 2.1 平台与被保护程序解耦

授权系统应独立于业务程序存在，业务程序只通过 SDK / API 接入，不把授权逻辑写死在单个项目中。

### 2.2 在线为主，离线为辅

主模型采用 **在线授权验证**，本地只保留短期租约和必要缓存。  
长期离线不作为默认能力，只作为企业客户特例。

### 2.3 控制权优先

商业软件最重要的是：

- 谁能用
- 用多久
- 用几个设备
- 能用哪些功能
- 是否能继续升级
- 是否能被远程停用

系统必须保证这些控制权在服务端，而不是放在客户端本地。

### 2.4 保护链分层

“授权系统”与“加壳/代码保护”不能混为一体，应该拆成三层：

1. 授权层
2. 本地保护层
3. 发行控制层

加壳属于可插拔增强能力，不应成为系统唯一依赖。

---

## 3. 产品边界

## 3.1 本系统负责

- License 生命周期管理
- 激活/续租/心跳/封禁
- 设备绑定
- 授权界面
- 本地票据加密存储
- SDK 封装
- 程序接入方式标准化
- 管理后台
- 授权分发与统计

## 3.2 本系统不直接负责

- 真正的商业壳实现
- PE 虚拟化
- VM 保护
- 深度反调试引擎
- 反逆向底层壳

这些应通过 **外部保护器适配接口** 接入，例如：

- SP
- WinLicense
- VMProtect
- Themida
- 自定义 Loader

## 5. 核心能力模块

## 5.1 授权模型

授权核心建议使用 **License + Device + Lease** 三段式模型。

### License

表示一份购买或分配的授权。

字段建议：

- `license_id`
- `license_key`
- `product_id`
- `plan_id`
- `status`
- `issued_at`
- `expire_at`
- `max_devices`
- `max_concurrency`
- `feature_flags`
- `customer_id`
- `channel_id`
- `notes`

### Device

表示具体激活到哪台机器。

字段建议：

- `device_id`
- `license_id`
- `device_fingerprint`
- `device_name`
- `os_info`
- `hardware_summary`
- `first_activate_at`
- `last_seen_at`
- `status`

### Lease

表示客户端当前拿到的短期使用租约。

字段建议：

- `lease_id`
- `license_id`
- `device_id`
- `issued_at`
- `expire_at`
- `nonce`
- `client_version`
- `signature`

设计原因：

- License 控制长期授权
- Device 控制设备绑定
- Lease 控制在线使用时长

这样做后，服务端可以随时：

- 禁用 License
- 移除某个设备
- 拒绝发放新 Lease

---

## 5.2 在线激活

### 流程

1. 用户打开程序
2. 如果本地无有效 Lease，则进入授权 UI
3. 输入 `License Key`
4. SDK 采集设备指纹
5. 服务端校验：
   - key 是否存在
   - 是否已到期
   - 是否被封禁
   - 设备数是否超限
   - 版本是否允许
6. 验证通过后：
   - 创建或绑定 Device
   - 下发短期 Lease
   - 下发功能位
7. 本地加密保存 Lease
8. 程序放行进入主功能

### 返回内容建议

- `license_status`
- `lease_token`
- `lease_expire_at`
- `feature_flags`
- `force_upgrade`
- `message`

---

## 5.3 在线验证与心跳

### 目标

不是只在首次激活时校验，而是持续校验。

### 机制

- 启动时校验一次
- 运行中每 10-30 分钟心跳一次
- 关闭关键功能前可按需追加验证

### 服务端检查项

- License 是否仍有效
- Device 是否被移除
- 是否超并发
- 当前版本是否被禁用
- 是否需要强制升级
- 是否需要降级功能位

### 本地宽限期

允许短时间断网，例如：

- 24 小时
- 72 小时
- 7 天

超过宽限期则必须联网续租。

## 5.4 授权 UI

系统应内置一套 **通用授权界面**，可嵌入到不同程序中。

### UI 组成

- 登录/激活页
- 设备占用提示页
- 到期提示页
- 续费提示页
- 强制升级提示页
- 离线授权导入页
- 激活记录与解绑页

### 风格要求

- 支持主题定制
- 支持 Logo / 品牌换肤
- 支持配置字段显示
- 支持内嵌或独立窗口模式

### 模式

#### 模式 A：内嵌 UI

适合：

- Electron
- .NET
- Qt

#### 模式 B：独立授权窗口

适合：

- 原生 C/C++ 程序
- 不方便内嵌 UI 的项目

---

## 5.5 本地缓存与加密

在线授权系统不能完全依赖每次联网，否则体验差。  
因此需要本地缓存，但缓存必须受保护。

### 本地保存内容

- Lease Token
- 最近一次成功验证时间
- 功能位快照
- 设备 ID

### 加密方式建议

- 使用对称加密存储票据
- 密钥不硬编码为明文
- 尽量结合：
  - 程序内置常量
  - 设备指纹
  - 时间因子
  - 原生层混淆

### Windows 建议

优先组合使用：

- DPAPI
- 本地加密文件
- 程序内二次封装

### 风险提示

本地缓存只能提高篡改成本，不能替代服务端控制。

---

## 5.6 设备绑定

### 目标

限制一份 License 被随意传播。

### 绑定策略建议

#### 严格绑定

适合高价值软件：

- 绑定设备指纹
- 更换关键硬件后视为新设备

#### 弹性绑定

适合大众软件：

- 容忍少量硬件变化
- 例如磁盘、网卡、主板特征组合打分

### 支持的动作

- 查看已激活设备
- 移除设备
- 限制设备数量
- 限制解绑次数
- 记录解绑日志

---

## 5.7 功能位与套餐

不能只有“能用/不能用”两档。

应支持按产品和套餐控制功能：

- 基础版
- 专业版
- 企业版
- 渠道版
- 定制版

### 功能位形式建议

- 布尔位
- 数值限制
- 并发限制
- 模块白名单

例如：

- `feature_publish = true`
- `feature_batch_login = false`
- `max_window_count = 20`
- `max_team_members = 3`

---

## 5.8 风控与审计

授权系统必须具备基础风控能力。

### 风控信号

- 同一 Key 在多个城市/公网 IP 高频切换
- 短时间大量激活/解绑
- 同一设备频繁改指纹
- 长期使用过期 Lease
- 非法版本号访问
- 异常心跳频率

### 审计日志

必须记录：

- 激活
- 验证
- 续租
- 解绑
- 到期
- 封禁
- 功能变更
- 强制升级

## 6. SDK 设计

## 6.1 SDK 分层

通用 SDK 建议拆成以下层：

### Core SDK

负责：

- 设备指纹
- License 激活
- Lease 验证
- Token 解密
- 本地缓存
- 心跳续租

### UI SDK

负责：

- 激活页
- 错误提示页
- 到期页
- 设备管理页

### Protection Adapter SDK

负责：

- 与加壳器/启动器/Loader 的衔接
- 启动前校验
- 本地环境检查

---

## 6.2 支持的接入方式

### Electron SDK

形态：

- Node 原生模块 + JS 封装
- preload API
- 可选独立授权窗口

### C/C++ SDK

形态：

- DLL
- 静态库
- 回调式 UI 接口

### .NET SDK

形态：

- NuGet 包
- WPF/WinForms UI 组件

### HTTP Only SDK

给不方便接原生库的程序提供纯 HTTP 接口模式，但安全性较弱。

## 7. 加壳 / 保护能力如何纳入系统

## 7.1 关键原则

授权系统本身不要内置真正的壳引擎。  
应设计成 **“保护适配层”**，兼容多种外部保护方案。

### 为什么

真正可用的商业壳研发成本极高，不适合自行从零实现。

## 7.2 保护能力应拆成三档

### 档 1：零成本保护

- JS 字节码
- 资源混淆
- 本地缓存加密
- 启动器校验
- 签名校验

### 档 2：低成本保护

- 自定义 Loader
- 原生启动器
- 关键逻辑 native 化
- 本地防篡改检测

### 档 3：外部商业保护器接入

通过适配器接入：

- SP
- WinLicense
- VMProtect
- Themida

系统只负责：

- 提供统一授权票据
- 提供启动前回调接口
- 提供 UI 和 License 流转

### 适配器接口建议

- `BeforeProgramStart()`
- `OnLicenseUIRequested()`
- `OnActivate()`
- `OnLeaseRefresh()`
- `OnDeviceBind()`
- `OnProtectionFailure()`

## 9. 产品模型

系统应从一开始支持多产品。

### 产品结构建议

- `Product`
  - 例如：软件 A / 软件 B / 软件 C

- `Edition / Plan`
  - 基础版 / 专业版 / 企业版

- `Version Policy`
  - 哪些版本允许继续使用
  - 哪些版本必须升级

- `Feature Policy`
  - 各版本功能位

这样后续每个程序只需要接同一套 SDK，然后在服务端按 `product_id` 区分即可。

---

## 10. API 设计建议

核心接口建议最少包括：

- `POST /activate`
- `POST /verify`
- `POST /heartbeat`
- `POST /deactivate`
- `POST /device/remove`
- `GET /license/info`
- `GET /feature-flags`
- `GET /upgrade-policy`

### 返回规范建议

- 统一状态码
- 统一错误码
- 明确用户可见消息与内部日志字段分离

例如：

- `LICENSE_EXPIRED`
- `DEVICE_LIMIT_REACHED`
- `LEASE_INVALID`
- `FORCE_UPGRADE_REQUIRED`
- `LICENSE_BANNED`

---

## 11. 安全设计重点

## 11.1 通信安全

- 全部使用 HTTPS
- 请求带时间戳
- 请求带 nonce
- 签名防重放
- 对关键接口限流

## 11.2 本地安全

- Token 本地加密
- 缩短 Lease 有效期
- 本地时间异常检测
- 程序版本签名校验
- 关键校验尽量在 native 层

## 11.3 服务端安全

- License 改动全审计
- 激活与设备变更需要记录操作者
- 高危操作支持二次确认

## 12. 建议的技术实现路线

## 12.1 MVP（第一版）

目标：尽快能接入多个程序。

### 范围

- 一个授权服务端
- 一个通用授权 UI
- 一个 Electron SDK
- 一个 C/C++ SDK
- 一个后台管理系统
- 在线激活 + 续租 + 设备绑定 + 功能位

### 保护能力

- 不做真正壳
- 先做本地票据加密 + Loader / native 辅助校验

增加：

- 多渠道
- 卡密 / 激活码
- 离线导入授权
- 更完整风控
- 自动升级策略
- 更多 SDK 语言

## 12.3 V3

增加：

- 商业壳适配器
- 插件化授权 UI
- 多租户后台
- 自助购买、支付、自动发卡

## 14. 商业模式建议

该授权系统本身可以作为独立产品卖：

- 按产品数收费
- 按活跃设备数收费
- 按 License 数收费
- 按渠道版收费
- 按私有化部署收费

如果只内部使用，也能作为公司统一授权平台，避免每个程序各自造轮子。

## 15. 最终结论

如果目标是：

> 做一套后续多个程序都能接入的、带 UI、在线授权、验证、加密和保护接入能力的通用授权系统

那么最合理的做法不是从“某个程序的壳”开始，而是建立一套：

- **统一授权平台**
- **统一 SDK**
- **统一授权 UI**
- **统一设备绑定与在线验证**
- **统一保护适配层**

其中：

- **授权、验证、设备绑定、功能位、后台管理** 是系统核心
- **加壳** 是可插拔增强能力
- **真正值钱的控制权必须在服务端**

这套思路才能支持后续多个程序重复接入，而不是每个项目单独重做一次。
