# 通用授权系统-全套开发计划

## 1. 文档目标

本文档基于当前目录下已有的产品、架构、数据库 API、WBS 与商业化保护方案文档，整理形成一份可直接用于排期、分工、开发、联调、测试和上线的完整开发计划。

本计划面向 **MVP 到 V1 阶段**，核心目标是先完成一套可商业化使用的通用在线授权系统，再逐步扩展风控、渠道、离线授权、更多 SDK 与保护器适配能力。

---

## 2. 项目定位

本项目定位为一套独立于具体业务程序的 **通用在线授权平台**，为多个桌面软件提供统一的授权、验证、设备绑定、功能位控制、后台管理和 SDK 接入能力。

系统最终由以下部分组成：

1. 授权服务端
2. 管理后台
3. 通用授权 UI
4. Electron SDK
5. C/C++ SDK
6. 数据库与缓存基础设施
7. 审计与风控模块
8. 可插拔保护适配层

核心原则：

- 服务端掌握最终授权控制权
- 客户端只保存短期可撤销 Lease
- 授权系统与加壳/本地保护解耦
- MVP 优先跑通授权商业闭环
- 后续再扩展渠道、离线授权、多租户和保护器适配

---

## 3. 产品目标

### 3.1 核心目标

建立一个可被多个桌面程序复用的统一授权平台，实现：

1. 用户可通过 License Key 在线激活软件
2. 软件可按设备、版本、功能位、到期时间进行控制
3. 平台可远程控制软件的授权状态和可用能力
4. 每个接入程序只需接入 SDK，不必重复开发授权逻辑
5. 管理员可通过后台统一管理产品、套餐、License、设备和日志

### 3.2 商业目标

- 支撑多个商业软件统一授权
- 降低新软件商业化接入成本
- 提升对盗版、共享、滥用的控制能力
- 为后续渠道、卡密、订阅、企业授权打基础

### 3.3 本期非目标

MVP 阶段不包含：

- 自研完整商业壳
- 深度虚拟化加壳引擎
- 完整支付系统
- 财务对账系统
- 多租户 SaaS 化
- 完整渠道/代理商体系
- 长期离线授权体系

## 7. 核心业务模型

### 7.1 Product

表示接入授权平台的产品。

关键字段：

- product_code
- name
- status
- description

### 7.2 Plan

表示产品下的套餐。

关键字段：

- product_id
- plan_code
- name
- duration_days
- max_devices
- max_concurrency
- grace_hours
- feature_flags

### 7.3 License

表示一份具体授权。

关键字段：

- license_key
- product_id
- plan_id
- status
- issued_at
- activate_at
- expire_at
- max_devices_override
- feature_flags_override

### 7.4 Device

表示绑定设备。

关键字段：

- license_id
- device_code
- fingerprint_hash
- device_name
- os_type
- os_version
- app_version
- status
- first_activate_at
- last_seen_at

### 7.5 Lease

表示客户端当前持有的短期使用租约。

关键字段：

- license_id
- device_id
- lease_token_id
- issued_at
- expire_at
- nonce
- status
- signature

设计原则：

- License 控制长期授权
- Device 控制设备绑定
- Lease 控制短期在线使用权
- 客户端不保存长期核心授权权力

## 阶段 2：公共授权 API MVP

周期：4-7 天

目标：跑通激活、验证、心跳、解绑的核心授权链路。

### 2.1 License 校验服务

任务：

- 按 license key 查询授权
- 校验产品匹配
- 校验状态
- 校验到期时间
- 加载套餐
- 合成功能位
- 处理 override 配置

交付物：

- LicenseService

### 2.2 设备绑定服务

任务：

- 创建设备
- 查询设备
- 判断设备是否已绑定
- 判断设备数是否超限
- 更新 last_seen_at
- 手动解绑设备
- 封禁设备

交付物：

- DeviceService

### 2.3 Lease 签发与校验

任务：

- 设计 Lease Token 结构
- 生成 lease token
- 设置 lease 过期时间
- 保存 lease 记录
- 校验 lease 签名
- 校验 lease 状态
- 支持 lease 吊销

交付物：

- LeaseService

### 2.4 激活接口

接口：

```text
POST /api/v1/license/activate
```

任务：

- 接收 productCode、licenseKey、device 信息
- 校验 License
- 校验设备数
- 创建或复用 Device
- 签发 Lease
- 返回功能位和版本策略

### 2.5 验证接口

接口：

```text
POST /api/v1/license/verify
```

任务：

- 校验本地 lease token
- 判断 License 是否仍有效
- 判断 Device 是否被移除或封禁
- 判断是否需要续租
- 返回当前功能位和版本策略

### 2.6 心跳接口

接口：

```text
POST /api/v1/license/heartbeat
```

任务：

- 校验 lease
- 刷新设备 last_seen_at
- 续租
- 同步功能位
- 同步版本策略
- 写入心跳日志

### 2.7 解绑接口

接口：

```text
POST /api/v1/license/deactivate
```

任务：

- 校验 License 与 Device
- 移除设备
- 吊销当前 Lease
- 写入审计日志

交付物：

- 授权 API 可本地调用
- activate / verify / heartbeat / deactivate 全链路跑通
- 基础错误码体系
- API 测试集

## 13. 关键风险与应对

### 13.1 控制权放在客户端

风险：

- 客户端被修改后绕过授权

应对：

- 服务端签发短期 Lease
- 客户端只缓存短期票据
- License 状态、封禁、功能位由服务端控制

### 13.2 设备指纹过严

风险：

- 用户更换硬件或系统后误伤

应对：

- 支持弹性评分
- 保存 hash，不保存敏感明文
- 后台支持人工解绑

### 13.3 设备指纹过松

风险：

- License 容易被多人共享

应对：

- 组合多项设备特征
- 记录异常变动
- 加入风控规则

### 13.4 过早投入商业壳

风险：

- 成本高、周期长、收益不确定

应对：

- 先做授权系统
- 再做本地代码硬化
- 最后通过 Protection Adapter 接入商业保护器

### 13.5 Electron 本地安全偏弱

风险：

- 本地逻辑容易被修改或调试

应对：

- 收紧 Electron 安全配置
- 保留 bytenode
- 启用 fuses
- 启用 ASAR Integrity
- 关键校验迁移到主进程或 native 层

## 14. 与本地保护方案的关系

授权系统不直接实现商业壳，但应为保护方案预留接口。

推荐分层：

1. 授权层
   - License
   - Device
   - Lease
   - Feature Flags
   - Heartbeat

2. 本地保护层
   - bytenode
   - ASAR Integrity
   - Electron fuses
   - 本地缓存加密
   - Native 辅助校验

3. 保护适配层
   - SP
   - WinLicense
   - VMProtect
   - Themida
   - 自定义 Loader

保护适配器预留接口：

```text
BeforeProgramStart()
ShowLicenseUI()
ValidateLocalLease()
RequestOnlineVerify()
ProtectionFailureHandler()
```

## 17. 结论

本项目应优先建设 **服务端授权控制权**，而不是优先投入商业壳或复杂本地保护。

MVP 的核心是：

- 一套授权服务端
- 一套管理后台
- 一套通用授权 UI
- 一套 Electron SDK
- 一套短期 Lease 授权机制
- 一套基础日志与封禁能力

完成这些后，系统就已经具备首个商业软件接入能力。后续再逐步扩展 C/C++ SDK、风控、渠道、离线授权和保护器适配。
