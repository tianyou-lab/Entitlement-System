# 通用授权系统-PRD

## 1. 文档信息

- 产品名称：通用在线授权系统
- 文档类型：PRD
- 目标版本：MVP / V1
- 适用范围：多产品、多客户端接入的统一授权平台

---

## 2. 产品背景

当前许多桌面软件在商业化过程中面临以下共性问题：

- 每个程序单独实现授权，重复开发严重
- 仅靠本地 Key 校验，容易被共享与绕过
- 无法控制设备数、并发数、到期时间、功能位
- 无统一后台查看激活、设备、订单、封禁情况
- 缺少统一授权 UI，接入成本高
- 缺少与本地保护器、加密方案、加壳方案的协同能力

因此需要建立一套独立于单个程序的 **通用授权系统**，为多个程序统一提供：

- 授权 UI
- 在线激活
- 在线验证
- 到期控制
- 设备绑定
- 功能位控制
- 封禁与风控
- SDK 接入能力
- 管理后台

---

## 3. 产品目标

## 3.1 核心目标

建立一个可被多个桌面程序复用的统一授权平台，实现：

1. 用户可通过 License Key 在线激活软件
2. 软件可按设备、版本、功能位、到期时间进行控制
3. 平台可远程控制软件的授权状态和可用能力
4. 每个接入程序只需接 SDK，不必重复造轮子

## 3.2 商业目标

- 支撑公司后续多个商业软件统一授权
- 降低每个新软件的商业化接入成本
- 提升盗版、共用、滥用的控制能力
- 为后续渠道、卡密、订阅、企业授权打基础

## 3.3 非目标

本期不作为目标的内容：

- 自研完整商业壳
- 深度虚拟化加壳引擎
- 跨平台移动端授权
- 完整支付系统
- 财务对账系统

## 5. 使用场景

## 5.1 终端用户场景

### 场景 A：首次激活

用户安装软件后首次打开，系统提示输入 License Key，验证通过后开始使用。

### 场景 B：到期续用

用户授权到期后打开软件，系统提示续费或更换授权。

### 场景 C：设备数超限

用户在新设备激活时，若超出最大设备数，提示释放旧设备或联系管理员。

### 场景 D：断网短期使用

用户在短期离线环境下，仍可使用上一次在线校验签发的短期租约。

### 场景 E：强制升级

软件版本被后台设置为必须升级时，客户端在验证后提示升级并限制继续使用。

## 5.2 平台管理场景

### 场景 F：创建 License

管理员在后台创建一批授权，指定产品、套餐、有效期、最大设备数。

### 场景 G：远程封禁

发现某授权被滥用，管理员在后台封禁该 License，客户端下一次验证时失效。

### 场景 H：调整功能位

客户升级套餐后，管理员调整功能位，客户端心跳后自动获得新能力。

### 场景 I：查看设备

客服可查看某个 License 当前绑定的设备，并协助客户解绑。

## 7. 功能清单

## 7.1 用户端功能

### 7.1.1 激活页

功能说明：

- 用户输入 License Key
- 查看产品/版本说明
- 提交激活

基础要求：

- 提示网络状态
- 显示错误原因
- 支持复制错误码

### 7.1.2 验证流程

功能说明：

- 启动时自动校验本地租约
- 租约不足时自动联网续租

### 7.1.3 设备占用提示页

- 可展示当前已绑定设备数
- 可提示联系管理员或进入自助解绑流程

### 7.1.4 到期提示页

功能说明：

- 展示剩余天数
- 展示续费提示
- 展示限制原因

### 7.1.5 强制升级提示页

功能说明：

- 显示当前版本已停用
- 引导下载新版本

## 9. 业务规则

## 9.1 授权状态

建议支持：

- `active`
- `inactive`
- `expired`
- `banned`
- `suspended`

## 9.2 激活规则

- 一个 License 可绑定多个设备，但不得超过 `max_devices`
- 每个设备绑定唯一指纹
- 超限时拒绝激活或要求移除旧设备

## 9.3 验证规则

- 客户端必须持有有效 Lease 才能使用主功能
- Lease 过期必须联网续租
- 服务端可拒绝续租

## 9.4 版本规则

- 后台可为产品配置最低允许版本
- 若当前版本低于最低版本，可要求强制升级

## 9.5 功能位规则

- 功能位由套餐决定
- 心跳成功后客户端应同步最新功能位

MVP 达标标准：

1. 支持创建多个产品
2. 支持创建套餐与功能位
3. 支持生成并管理 License
4. 支持客户端在线激活

---

## 13. 风险与约束

### 风险

- 本地缓存如果设计过弱，容易被篡改
- 设备指纹过于严格会误伤正常用户
- 设备指纹过于宽松会导致共享泛滥
- 仅靠客户端校验容易被绕过

### 约束

- 核心控制权必须留在服务端
- 离线能力不能削弱整体控制力
