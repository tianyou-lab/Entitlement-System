# 零到低成本的商业化保护方案文档

## 1. 目标

本文档面向当前仓库 `by-chrome-qihao-0418`，目标是在 **不购买高价壳/授权产品** 的前提下，为本套 Electron 自动化程序建立一套 **可上线销售、可运营、可逐步增强** 的商业化保护方案。

本方案默认约束如下：

- 保持当前主程序为 `x64` 发行线，不强行改造为 `ia32`
- 不采购 WinLicense / VMProtect / Themida / SP 等高价商用保护器
- 允许使用开源工具、免费能力、自建服务
- 保护目标不是“绝对不可破解”，而是“显著提高破解与滥用成本，并具备持续运营能力”

## 2. 商业化保护的真实目标

对这类 Electron 自动化软件，真正需要保护的是以下三类资产：

1. **授权收益**
   - 防止一份许可多人共用
   - 防止到期后继续长期使用
   - 防止被批量转卖

2. **核心业务逻辑**
   - 防止值钱的规则、策略、风控逻辑被完整拿走
   - 防止最核心的算法被直接复用

3. **运行控制权**
   - 能封禁滥用用户
   - 能控制功能开关
   - 能控制更新、灰度、停服和强制升级

结论：对于商业软件，**授权体系和云端控制权** 的优先级高于单纯“加壳”。

## 3. 当前仓库现状判断

当前项目已经有一些基础保护，但仍存在明显短板。

### 已有基础

- 已使用 `bytenode` 字节码编译
- 已有 `build:secure` / `compile:bytecode` 保护构建链
- 已存在主进程完整性校验
- 已拦截部分调试参数
- 已有原生模块 `window-api`

### 当前明显短板

- `BrowserWindow.webPreferences.webSecurity = false`
- `sandbox = false`
- 生产环境 CSP 仍允许 `unsafe-inline` / `unsafe-eval`
- 仍使用 `file://` 直接加载 renderer
- 还没有完整的商业授权体系
- 还没有设备绑定、心跳、封禁、功能位控制
- 关键业务逻辑仍主要在本地
- 缺少统一的升级门禁与分发控制

### 风险判断

如果现在直接卖软件，最容易被利用的路径不是“深度逆向 native 模块”，而是：

- 拿到安装包后长期离线使用
- 共用账号 / 共用许可
- 修改本地校验逻辑绕过简单授权
- 抓取 API 逻辑并伪造调用
- 直接复用客户端界面和流程

## 4. 总体方案

建议采用三层保护：

### 第一层：本地代码硬化

目标：提高静态分析和随手修改成本。

手段：

- 保留 `bytenode`
- renderer 做适度混淆
- 打开 Electron fuses
- 启用 ASAR Integrity
- 改善 Electron 安全配置
- 保留原生模块承载部分敏感逻辑

### 第二层：授权体系

目标：把“能不能用、能用多久、能用哪些功能”的控制权掌握在自己手里。

手段：

- 自建 license 服务
- 设备绑定
- 激活/反激活
- 到期控制
- 功能位控制
- 心跳续租
- 黑名单/封禁

### 第三层：云端核心

目标：把真正值钱的决策逻辑尽量留在服务端。

手段：

- 风控规则上云
- 任务审批上云
- 高价值策略上云
- 更新门禁上云
- 下载分发上云

## 5. 分阶段实施路线

---

## 阶段 0：发行基线调整（1-2 天）

### 目标

先把发行方式切到更适合商业化的路线，为后续授权和升级打基础。

### 建议

- 默认发行 `NSIS` 安装包或 `--dir` 目录包
- 暂时不要把 `portable` / `single-exe` 作为主发行线
- 所有安全增强先在 `x64` 路线上完成

### 原因

- 安装版更容易做更新、写入本地安全目录、注册服务、保存许可状态
- 单文件便携版更容易被随手拷走和批量分发
- Electron + native 模块 + 单文件打包组合更难稳定加固

### 交付物

- 一个稳定的 `build:win` 发行基线
- 一个稳定的 `build:unpack` 测试基线

---

## 阶段 1：本地代码硬化（3-5 天）

### 1.1 修正 Electron 安全项

优先修改以下内容：

- `src/main/index.js`
  - 把 `webSecurity` 调回 `true`
  - 评估是否能启用 `sandbox: true`
  - 缩减 CSP 中的 `unsafe-inline` 和 `unsafe-eval`
  - 逐步摆脱 `file://`，改为自定义协议或更严的资源加载方式

### 1.2 打开 Electron Fuses

建议启用：

- 禁止 `RunAsNode`
- 禁止 `NODE_OPTIONS`
- 禁止 CLI Inspect 参数
- 只允许从 ASAR 启动
- 启用嵌入式 ASAR 完整性校验

### 1.3 启用 ASAR Integrity

在 `electron-builder` 发行链中启用 ASAR 完整性，并配合 “Only load app from ASAR” 使用。

### 1.4 保留 `bytenode`

继续对以下部分使用字节码保护：

- `src/main`
- `src/preload`

原则：

- 业务核心优先保护主进程与 preload
- renderer 只保护高价值页面逻辑，不要无差别全量重混淆

### 1.5 renderer 做适度混淆

建议：

- 只对关键页面、关键配置、关键字符串启用混淆
- 避免所有页面都开最高级别混淆

### 1.6 原生模块职责增强

当前已有 `window-api` 原生模块。后续可把少量本地敏感逻辑继续放到 native 层，例如：

- 本地授权票据签名校验
- 心跳 nonce 处理
- 反重放辅助校验
- 启动环境特征采集

注意：native 模块只是增大分析成本，不等于绝对安全。

### 阶段 1 交付物

- 修正后的 `src/main/index.js`
- fuses 配置
- ASAR Integrity 配置
- 保留并验证 `bytenode` 构建链
- 一版稳定的 `x64` 安装包

## 阶段 2：自建授权系统（5-10 天）

### 2.1 授权模型

建议最小可用模型如下：

#### License 表

- `license_key`
- `plan_code`
- `status`
- `expire_at`
- `max_devices`
- `max_concurrency`
- `feature_flags`
- `created_at`
- `updated_at`

#### Device 表

- `license_id`
- `device_id`
- `device_fingerprint`
- `first_activate_at`
- `last_seen_at`
- `status`

#### Lease / Heartbeat 表

- `license_id`
- `device_id`
- `lease_token_id`
- `lease_expire_at`
- `last_heartbeat_at`
- `client_version`
- `ip`

### 2.2 授权流程

#### 首次激活

1. 用户输入 license key
2. 客户端采集设备指纹
3. 服务端校验 key 是否存在、是否到期、是否超设备数
4. 通过后签发一个 **短期 lease token**
5. 客户端本地保存 token 与必要的缓存状态

#### 正常启动

1. 客户端先读本地 token
2. 验证签名与有效期
3. 在允许的宽限期内联网做一次续租
4. 服务端返回新的 lease token

#### 心跳

建议每 10-30 分钟做一次：

- 检查 license 状态
- 检查设备是否被移除
- 检查版本是否被强制升级
- 检查功能位是否变化

### 2.3 设备指纹建议

不要依赖单一硬件参数。建议组合：

- Windows 机器名
- 系统盘卷标 / Volume Serial
- 主板 / BIOS 标识（可选）
- CPU 基本信息（可选）
- 当前系统 SID / 用户环境特征（谨慎）

策略：

- 生成稳定指纹摘要
- 不存原始敏感值，尽量存哈希
- 允许少量硬件漂移

### 2.4 离线策略

建议默认在线授权，不做永久离线。

可以给企业客户提供：

- 7 天 / 15 天 / 30 天离线租约

不要给普通客户长期离线 license，否则控制力会明显下降。

### 2.5 功能位控制

不要只返回一个布尔值“是否授权”。

建议按功能位返回：

- `multi_open_limit`
- `proxy_feature`
- `publish_feature`
- `batch_login_feature`
- `team_feature`
- `cloud_task_feature`

这样即使用户共享 key，也能按套餐限制价值。

### 阶段 2 交付物

- license 服务端 API
- 设备绑定逻辑
- 心跳续租逻辑
- 功能位控制逻辑
- 主进程启动前授权校验逻辑

---

## 阶段 3：云端核心迁移（持续迭代）

### 目标

把最值钱的东西从客户端挪到服务端，降低“客户端被拿走 = 核心能力被拿走”的风险。

### 优先迁移对象

#### 第一优先级

- 套餐权限判断
- 风控规则
- 黑名单
- 下载授权
- 更新门禁

#### 第二优先级

- 高价值任务审批
- 账号质量评分
- 特定平台适配逻辑
- 敏感策略参数

#### 第三优先级

- 自动化调度策略
- 高阶模板和策略库

### 原则

- 本地保留执行层
- 服务端保留决策层
- 客户端不拿完整规则，只拿最终决策结果

---

## 阶段 4：发行与运维控制（3-5 天）

### 4.1 更新门禁

建议所有新版本下载都与 license 状态绑定：

- 未授权：不给下载
- 已过期：不给下载
- 被封禁：不给下载
- 指定计划：仅允许下指定版本

### 4.2 远程开关

建议服务端保留以下开关：

- 强制升级
- 全局暂停某功能
- 下线某版本
- 拉黑某批设备
- 灰度启用某功能

### 4.3 风险监控

需要记录：

- 同一 license 的设备数
- 异常频繁激活/反激活
- 异地高频登录
- 心跳异常
- 频繁修改系统时间
- 客户端版本分布

### 4.4 审计

建议把以下行为入库：

- 激活
- 续租
- 到期
- 封禁
- 设备移除
- 升级检查

## 7. 成本估算

### 零成本版本

可先完成：

- 修 Electron 安全项
- 保留 `bytenode`
- 启用 fuses
- 启用 ASAR Integrity
- 自建基础 license 服务

成本：

- 现金成本：约 0
- 时间成本：约 2-3 周

### 低成本版本

增加：

- 一台云服务器
- 域名
- HTTPS 证书
- 简单日志与告警

成本：

- 现金成本：约 50-300 RMB / 月
- 时间成本：约 3-5 周

### 中成本版本

增加：

- 商业代码签名
- 更完整的监控
- 更细粒度套餐

成本：

- 现金成本：约 1000+ RMB / 年起

---

## 8. 不要做的事

- 不要把所有核心规则都留在本地
- 不要只做一个“本地 license 文件 + 本地校验”
- 不要把便携版作为唯一商业发行版
- 不要在 Electron 安全项明显偏弱时先去纠结壳
- 不要期望前端混淆本身就能形成商业壁垒

---

## 9. 推荐优先级

按投入产出比排序，建议这样做：

1. 修 Electron 安全配置
2. 保留 `bytenode`，完善构建链
3. 启用 fuses + ASAR Integrity
4. 自建 license 服务
5. 做设备绑定、心跳、功能位
6. 把关键规则逐步上云
7. 做更新门禁和远程开关
8. 后期再评估是否要上商业壳

---

## 10. 最终结论

对于当前这套 Electron 自动化程序，**零到低成本最优解不是买壳，而是：**

- **本地代码硬化**
- **服务端授权体系**
- **关键业务逻辑上云**
- **发行与更新控制**

这套方案虽然不等于“绝对防破解”，但已经足够支撑商业化上线，并且能随着收入增长逐步增强。

如果后续预算允许，再考虑把其中某一层升级为商用方案，例如：

- 商业代码签名
- 商业授权平台
- 商业 native 保护器

在此之前，先把 **控制权** 建起来，比先买壳更重要。
