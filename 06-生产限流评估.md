# 生产限流评估

更新时间：2026-05-14

## 1. 当前实现

公共客户端 API 已启用轻量内存限流：

- `/api/v1/license/*`
- `/api/v1/version/*`
- `/api/v1/offline-packages/*`
- `/api/v1/device-unbind-requests`

默认策略：

- 维度：客户端 IP + 接口路径。
- 窗口：`PUBLIC_API_RATE_LIMIT_WINDOW_MS=60000`。
- 次数：`PUBLIC_API_RATE_LIMIT_MAX=120`。
- 超限错误码：`RATE_LIMITED`。

## 2. 当前结论

单实例试运行阶段可以继续使用内存限流。它实现简单、无外部依赖，足以拦截明显的误调用和低成本刷接口。

多实例部署前必须升级为集中式限流。原因是每个 Server 实例的内存计数互不共享，请求经负载均衡分散后，实际限流阈值会随实例数放大。

## 3. Redis 升级触发条件

满足任一条件时应接入 Redis 或网关层限流：

1. Server 副本数大于 1。
2. 授权 API 暴露在公网且有稳定客户流量。
3. 出现频繁激活、心跳异常或撞库式请求。
4. 需要按租户、产品、License、设备维度限流。
5. 需要运维侧实时查看限流命中数量。

## 4. 推荐方案

优先级从高到低：

1. 网关层限流：Nginx、Traefik、Cloudflare、API Gateway。
2. Redis 集中式滑动窗口或固定窗口限流。
3. 当前内存限流作为单实例兜底。

服务端 Redis 方案建议：

- key：`rate:{ip}:{method}:{path}`。
- value：窗口内计数。
- TTL：窗口长度。
- 原子操作：`INCR` + 首次设置 `PEXPIRE`，或 Lua 脚本。
- 后续可扩展 key：`rate:{tenant}:{product}:{license}:{path}`。

## 5. A 阶段处理决定

A 阶段暂不引入 Redis 依赖，保持 Docker Compose 简洁，降低试运行复杂度。

必须保留：

- 当前内存限流。
- 环境变量可调阈值。
- e2e 覆盖超限返回。

B 阶段或多实例部署前再落地 Redis/网关限流，并补充：

- Redis 容器或托管 Redis 配置。
- 限流指标和告警。
- 按租户/产品维度的运营报表。
