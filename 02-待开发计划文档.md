# 待开发计划文档

更新时间：2026-05-14

## 1. 当前项目阶段

项目已经完成通用授权系统的 MVP 到准生产版本主体功能，当前阶段不是继续补核心链路，而是进入：

1. 真实桌面客户端验收。
2. 生产上线前加固。
3. SDK 制品化和客户接入体验完善。
4. 商业化、SaaS 化和运营能力扩展。

一句话结论：核心授权闭环已经打通，后台运营能力已经可用，下一步应优先把系统从“可运行”推进到“可交付、可监控、可回滚、可给客户接入”。

## 2. 已完成进度

### 2.1 核心授权链路

已完成：

- 产品、套餐、授权卡密、设备绑定、License 状态管理。
- 在线激活、验证、心跳续租、解绑、授权信息查询。
- Lease token 签发、验证、续租和过期处理。
- 设备数限制、封禁、暂停、过期和强制升级。
- 统一响应结构和业务错误码。

当前判断：P0 核心链路已完成，可进入真实客户端联调。

### 2.2 管理后台

已完成：

- 管理员登录、JWT 鉴权、首次登录强制改密。
- 运营控制台和独立模块切面。
- 产品管理、套餐配置、授权卡密、设备绑定、版本策略、风控面板、渠道与卡密、离线与解绑、保护器适配、运行日志。
- 运行日志支持下拉选择激活日志、心跳日志、审计日志。
- SDK 接入模块，提供源码 ZIP、Electron SDK、C++ SDK Demo、.NET SDK、License UI 入口和接入说明。
- 后台 UI 已做现代化视觉优化。

当前判断：后台已能支持内部运营，但权限、审计深度、批量操作安全和监控能力仍需生产化增强。

### 2.3 SDK 与客户端

已完成：

- Electron SDK：激活、验证、心跳、解绑、本地加密缓存、请求签名。
- Electron Demo：具备桌面接入示例，仍需真实 GUI 人工验收。
- License UI：可嵌入授权激活流程。
- C++ SDK Demo：具备基础客户端示例。
- .NET SDK：激活、验证、心跳、解绑、版本策略接口和 HMAC 请求签名。

当前判断：SDK 源码和 Demo 已具备接入基础，但还没有正式版本化制品、包管理发布和完整客户接入文档集。

### 2.4 安全和部署

已完成：

- 生产 seed 必须提供 `ADMIN_PASSWORD`。
- 管理员首次登录必须修改默认密码。
- 公共 API 支持 HMAC 请求签名。
- nonce 防重放和时间窗口校验。
- 公共 API 已增加轻量内存限流，默认每客户端 IP、每接口路径每分钟 120 次，可通过环境变量调整。
- 已增加独立 `/health` 健康检查接口和 Docker Compose healthcheck。
- 已初始化标准 Prisma migration 目录，生产迁移脚本执行 `prisma migrate deploy`。
- Dockerfile、Docker Compose、生产环境变量样例。
- GitHub Actions CI 覆盖 Server、Admin、License UI、Electron SDK/Demo、C++、.NET 和 Docker image build。
- Docker 镜像 artifact 工作流已存在。

当前判断：已有基础安全措施和 CI，但生产环境仍缺少监控、集中式限流、迁移治理、备份演练和正式发布流程。

## 3. 当前主要缺口

### 3.1 上线前必须补齐

| 编号 | 缺口 | 影响 |
|---|---|---|
| G1 | Electron GUI 真实桌面验收未完成 | 无法确认最终用户客户端体验 |
| G2 | .NET SDK 本机环境验收未完成 | Windows/.NET 客户端交付仍有风险 |
| G3 | migration 流程已初始化，但仍缺少变更审核和回滚演练制度 | 生产升级仍需治理 |
| G4 | 缺少生产监控和告警 | 出问题后发现慢、定位慢 |
| G5 | 限流仍是单实例内存策略，未接入 Redis 等集中式限流 | 多实例部署时限流统计不共享 |
| G6 | 后台权限仍是基础管理员模型 | 多运营人员使用时权限边界不足 |
| G7 | SDK 下载仍指向源码，没有正式版本制品 | 客户接入和版本回滚不够专业 |

### 3.2 可后置但应规划

| 编号 | 缺口 | 影响 |
|---|---|---|
| G8 | 订单、支付、自动发码未实现 | 还不是商业闭环系统 |
| G9 | SaaS 租户隔离仍是基础模型级 | 不能直接开放多租户自助运营 |
| G10 | 离线授权包导出/导入体验不完整 | 离线客户交付成本较高 |
| G11 | 保护器适配仍是登记模型 | 尚未和打包/加壳流水线打通 |
| G12 | SDK 缺少跨语言一致的错误处理和本地缓存策略 | 客户端接入体验不一致 |

## 4. 下一步开发路线

### 阶段 A：上线前验收和稳定性加固

目标：确认当前 MVP 在真实环境可以交付试运行。

建议周期：1-2 周。

任务：

1. 在桌面环境完成 Electron Demo 人工验收。
2. 在 Windows 或具备 .NET 8 SDK 的环境完成 .NET SDK Demo 验收。
3. 建立 Prisma migration 变更审核、演练和回滚制度。
4. 补充基础日志查看脚本和生产健康检查巡检流程。
5. 增加生产备份和恢复演练文档。
6. 增加剩余关键接口 e2e 覆盖：Admin 审核解绑、离线包撤销、风险事件处理。
7. 评估多实例部署时是否需要 Redis 集中式限流。

验收标准：

- Electron Demo 可完成激活、重启缓存、心跳续租、封禁失效、强制升级提示。
- .NET Demo 可完成激活、验证、心跳和请求签名。
- CI 全绿。
- 新环境可按 README 和部署文档完整启动。
- 数据库迁移有可重复执行流程。

### 阶段 B：生产运营能力

目标：让后台适合多人运营和问题排查。

建议周期：2-4 周。

任务：

1. 管理员角色权限细分。
2. 高风险操作二次确认和操作原因记录。
3. 后台审计日志增强：登录、创建、修改、封禁、导出、审核。
4. 管理后台搜索、筛选、分页和批量操作增强。
5. 监控指标：
   - API 请求量、失败率、延迟。
   - 激活失败率、设备超限、封禁命中、强制升级命中。
   - 风险事件数量和未处理高危风险。
   - PostgreSQL 连接数、慢查询、容量。
6. 告警策略：服务不可用、高错误率、数据库异常、高危风险积压。

验收标准：

- 不同角色只能访问授权范围内的后台功能。
- 关键操作可追踪到操作者、时间、对象、变更内容和原因。
- 生产问题能通过日志和指标定位。

### 阶段 C：SDK 制品化和客户接入

目标：从源码示例升级为可交付 SDK。

建议周期：2-3 周。

任务：

1. GitHub Release 自动打包 SDK 制品。
2. Electron SDK 发布 npm 包或内部 tarball。
3. .NET SDK 发布 NuGet 包或离线 nupkg。
4. C++ SDK 发布按平台区分的 zip 包。
5. 后台 SDK 接入页从源码链接升级为版本化下载列表。
6. 每个 SDK 补充最小接入 README：
   - 安装方式。
   - 初始化配置。
   - 激活流程。
   - 启动验证流程。
   - 心跳续租流程。
   - 错误码处理。
   - 请求签名配置。
7. SDK 错误模型统一。

验收标准：

- 客户可在不阅读服务端源码的情况下完成 SDK 接入。
- 每个 SDK 有版本号、变更记录和可下载制品。
- 后台 SDK 页能明确展示最新版本、下载地址和接入步骤。

### 阶段 D：商业闭环

目标：把授权系统扩展为可售卖、可结算、可自动发货的商业系统。

建议周期：4-8 周。

任务：

1. 订单模型。
2. 支付回调。
3. 支付成功自动发放授权卡密。
4. 退款后自动禁用 License。
5. 续费、升级、降级。
6. 渠道价格策略。
7. 渠道销售统计和佣金结算。
8. 对账、导出、发票字段预留。

验收标准：

- 可从订单到授权卡密自动发放形成闭环。
- 渠道销售和结算数据可追踪。
- 退款、封禁和过期策略自动生效。

### 阶段 E：SaaS 化和高级安全

目标：支持多租户平台化运营和更高安全等级。

建议周期：持续迭代。

任务：

1. 所有后台查询按 tenant 隔离。
2. 管理员绑定 tenant 上下文。
3. 超级管理员跨租户管理。
4. 租户套餐、配额和到期时间。
5. 租户级品牌配置和密钥配置。
6. HMAC 密钥轮换机制。
7. IP 黑名单/白名单。
8. 设备指纹策略增强。
9. 管理后台 HTTPS、安全响应头、生产日志脱敏。

验收标准：

- 租户之间数据不可互相访问。
- 每个租户可独立配置授权策略和密钥。
- 安全策略可审计、可轮换、可回滚。

## 7. 建议验收清单

### 7.1 核心授权验收

- 使用 `DEMO-AAAA-BBBB-CCCC` 激活成功。
- 重启客户端后可读取本地缓存。
- 心跳后 lease token 可刷新。
- 后台封禁 License 后客户端再次验证失败。
- 后台移除设备后客户端再次验证失败。
- 后台开启强制升级后低版本客户端收到升级提示。

### 7.2 安全验收

- 未签名请求在生产签名开启时被拒绝。
- 重放 nonce 被拒绝。
- 过期 timestamp 被拒绝。
- 默认管理员首次登录必须改密。
- 非管理员接口无 token 被拒绝。
- 高风险操作有审计日志。

### 7.3 部署验收

- 新机器可通过 README 完整启动。
- Docker Compose 可启动 server/admin/license-ui/postgres。
- 数据库可备份和恢复。
- CI 全绿。
- 发布镜像和回滚流程可执行。

## 9. 结论

当前项目已经不处于“从零开发核心功能”的阶段，而是处于“准生产完善和交付化”阶段。下一步不建议继续堆新功能，优先级应放在真实桌面验收、生产迁移和限流、监控告警、权限审计、SDK 版本化制品上。等这些完成后，再进入订单支付、渠道结算和 SaaS 多租户平台化。
