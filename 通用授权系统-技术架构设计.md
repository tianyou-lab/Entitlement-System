# 通用授权系统-技术架构设计

## 1. 目标

本文档描述通用在线授权系统的技术架构设计，目标是支撑：

- 多产品接入
- 多客户端类型接入
- 在线激活、验证、心跳、设备绑定
- 后台管理、风控、日志审计
- 后续可接入加壳/保护适配器

---

## 2. 架构原则

### 2.1 平台化

授权系统必须独立于业务程序存在，业务程序通过 SDK / API 接入。

### 2.2 服务端为控制核心

授权状态、设备状态、功能位、强制升级、封禁等都由服务端控制。

### 2.3 SDK 跨程序复用

SDK 分层设计，适配不同客户端类型，而不是为单个程序硬编码。

### 2.4 可插拔保护层

加壳、启动器、Loader、原生保护方案应通过适配器接入，而不是耦合在授权核心内。

---

## 4. 逻辑分层

## 4.1 服务端分层

### 网关层

负责：

- 接入统一 API
- 鉴权
- 限流
- 日志打点
- 统一错误码处理

### 授权核心层

负责：

- License 状态判断
- 激活逻辑
- 验证逻辑
- Lease 签发逻辑
- 功能位分发

### 设备层

负责：

- 设备注册
- 设备绑定
- 设备移除
- 设备风控

### 风控与审计层

负责：

- 异常行为识别
- 关键操作记录
- 审计检索

### 后台管理层

负责：

- 管理员操作
- 产品配置
- License 运维
- 设备运维

## 4.2 客户端分层

### SDK Core

负责：

- 本地缓存
- Token / Lease 校验
- 设备指纹采集
- API 请求
- 错误处理

### SDK UI

负责：

- 激活 UI
- 到期提示 UI
- 升级提示 UI

### Protection Adapter

负责：

- 与壳 / Loader / 启动器集成
- 启动前拦截
- 票据二次校验

## 5. 服务端架构组件

## 5.1 API Gateway

建议职责：

- 暴露公共 API
- 统一鉴权中间件
- 请求签名验证
- 时间戳与 nonce 校验
- 接口限流

## 5.2 License Service

职责：

- 创建 License
- 校验 License 状态
- 绑定产品 / 套餐
- 返回功能位

## 5.3 Activation Service

职责：

- 首次激活
- 设备绑定
- 判断设备数上限
- 生成 Lease

## 5.4 Verify / Heartbeat Service

职责：

- 校验 Lease
- 续租
- 同步功能位
- 下发强制升级策略

## 5.5 Device Service

职责：

- 保存设备信息
- 移除设备
- 设备状态更新

## 5.6 Audit Service

职责：

- 记录关键行为
- 提供可查询审计记录

## 5.7 Risk Service

职责：

- 同一 License 多地异常使用识别
- 频繁解绑/激活识别
- 异常指纹变动识别

---

## 6. 客户端 SDK 架构

## 6.1 通用 SDK 模块

### Config

配置：

- API 地址
- Product ID
- 超时
- 宽限期

### Fingerprint

负责：

- 采集设备指纹
- 生成稳定摘要

### Store

负责：

- 本地持久化缓存
- 加密存储 Lease

### Crypto

负责：

- Token 解密
- 本地签名校验

### Client

负责：

- 调用 `/activate`
- 调用 `/verify`
- 调用 `/heartbeat`

### Feature Gate

负责：

- 解析并缓存功能位
- 向业务程序暴露授权态

## 6.2 Electron SDK

建议结构：

- `main` 侧授权核心
- `preload` 暴露只读授权状态
- `renderer` 内嵌授权 UI
- 可选 native addon 做本地票据辅助校验

## 6.3 C/C++ SDK

建议结构：

- 一个核心 DLL
- 一个回调式 UI 接口
- 提供：
  - `Activate`
  - `Verify`
  - `Heartbeat`
  - `GetFeatureFlags`

## 7. 票据设计

## 7.1 Lease Token

建议使用短期租约而不是长期永久 License 文件。

包含：

- `license_id`
- `product_id`
- `device_id`
- `feature_flags`
- `issued_at`
- `expire_at`
- `nonce`
- `signature`

## 7.2 本地存储策略

Windows 建议：

- 主缓存文件加密
- 可结合 DPAPI
- SDK 再做应用层二次封装

原则：

- 本地只存短期可撤销票据
- 不存长期核心控制权

---

## 8. 设备指纹策略

## 8.1 采集原则

- 不依赖单一特征
- 不存明文敏感信息
- 可容忍小幅硬件漂移

## 8.2 建议组成

- 机器名
- 卷序列号
- 主板/BIOS 基础标识
- CPU 基础标识
- 系统信息摘要

## 8.3 指纹算法策略

建议支持：

- 严格模式
- 弹性评分模式

不同产品可选不同策略。

---

## 9. 安全设计

## 9.1 通信安全

- HTTPS
- 服务端时间戳校验
- nonce 防重放
- API 签名
- 必要接口限流

## 9.2 客户端安全

- 缩短 Lease 生命周期
- 本地缓存加密
- 时间回拨检测
- 版本签名检查

## 9.3 服务端安全

- 管理后台 RBAC
- 高危操作审计
- License 变更记录

## 14. 与加壳/保护器的关系

授权系统与加壳系统解耦。

建议通过 `Protection Adapter` 实现：

- `BeforeProgramStart`
- `ShowLicenseUI`
- `ValidateLocalLease`
- `RequestOnlineVerify`
- `ProtectionFailureHandler`

这样未来可接：

- SP
- WinLicense
- VMProtect
- 自定义启动器

---

## 16. 结论

本架构的核心是：

- 服务端掌控授权状态
- SDK 统一接入多个程序
- UI 标准化
- 保护能力插件化

它能保证这套系统未来不是“给某一个程序做授权”，而是“成为多个程序共享的授权基础设施”。
